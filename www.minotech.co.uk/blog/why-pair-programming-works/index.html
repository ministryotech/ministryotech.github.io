Why pair programming works

By Keith Jackson 14/02/2011 Horsepower

Sometimes we all right bad code or, if not bad code, code that seems to have lost all reason somewhere down the line. A second pair of eyes is essential to stop us from making these mistakes. I have added a task to my board today to remove some methods from the Framework Extensions project (slimming it down and removing erroneous fluff is one of my main objectives with it at the moment) after someone replied to my docs post on the ReflectionHelper. They kindly pointed out that the Construct() method was, in fact, utterly pointless. They pointed out, quite rightly, that the following code will do the same thing for less...

MyObject o = Activator.CreateInstance();

I've used this method many times (perhaps more recently than when I wrote the original code) but code you know too well can have a tendency to become warm and comfortable like an old friend. You walk the danger line of defending it even when it's bad and even though you wouldn't write it that way now it still looks 'right' because you've seen it all before. This is where the pairing strategy comes in. I started pairing properly about 2 years ago and found it to be extremely effective. The only issue I took was that the commenter didn't leave their contact details; the first thing you learn when pairing for a while is to get used to being challenged all the time because without challenge there is no growth.

Anyone want to take on any code review duties on the framework code?
