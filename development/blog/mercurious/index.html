Mercurious

By Keith Jackson 07/07/2011 Propellant

I haven't written for a while as I've been extremely busy investigating new technologies for setting up a new development team infrastructure. This work is now well underway and I wanted to share some of my findings with you, well one in particular, Mercurial.

What is Mercurial?

Mercurial is a Source Control system, similar to Subversion or TFS (or SourceSafe if you're still repairing that!) As a part of my early investigations into setting up a technology stack for the development team I wanted to revisit all of the assumptions that I would have previously made from experience over tooling for everything from testing to source control to plugins for Visual Studio. I'm not going to lie to you, I never really like Subversion - I always found it a bit like a nagging sports injury that you learn to work around so it hurts you as little as possible as your experience with it grows. I'm not knocking Subversion too much though as there wasn't anything better - until now.

Mercurial and Git (not covered here in detail) are both DVCS (Distributed Version Control Systems). The joy of this is that both the working copy and the source control repository are on your local machine. This is in direct contrast to subversion where the repository only exists on the server. This allows developers to check in work on their local machines as and when they want without breaking the build for everyone else (Yes, Dave, you can put away the build hat now!). Annoyingly Subversion would often require you to check in a partial code fix, temporarily causing a break, particularly when doing heavy file renaming. You do learn ways around this as time goes on (wound analogy above) but it's not pleasant. What a DVCS introduces is two new concepts; whereas Commits & Updates are performed on your local repository, in order to submit changes for everyone else you have the concept of pulling and pushing to a central repository. In fact, with Mercurial, you can have any number of repositories in this chain so you could have a site based repository and a large central repository for distributed teams.

Size and Merge quality are other benefits which are made possible by the different nature of storage in Mercurial. Whereas Subversion is a file based storage system, Mercurial stores changesets and calculates the state of a given file. This also makes branching a breeze. Also, because a copy of the whole repository is available locally it all sits nicely in one file rather than having pesky '.svn' files in every single directory.

Oh, and it's all still free (Microsoft what are you playing at!)

Still Loving Tortoise

For users coming from Subversion or CVS, the transition is much easier as there is a TortoiseHg Mercurial client (Hg is the symbol for Mercury). This is very similar to TortoiseSVN but with the new options attached.

Free Repositories!

Mercurial is now the TFS alternative of choice on Codeplex and recently BitBucket have started to offer free private Mercurial hosting for groups of up to five users as well as free open source project hosting. I have just recently moved the MOT Framework Extensions first from Codeplex TFS to Codeplex Mercurial and now over to BitBucket. Bug tracking and Wiki services are also offered as part of the package. SO, if you're looking to play with something new or you are soon to be starting a new project try Mercurial, you really won't look back.
